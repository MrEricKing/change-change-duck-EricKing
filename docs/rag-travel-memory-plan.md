# RAG 旅行记忆增强计划

## 1. 背景与目标

小组项目「路线改改鸭」已经完成了从短视频到旅行路线的核心闭环：

```text
旅行视频 / 分享文案 -> ASR + OCR + 关键帧理解 -> LLM 规划 -> 地图路线 + 攻略卡片
```

个人作业的增量方向不是重新做一个旅行规划器，而是在现有项目基础上加入「旅行记忆」：

```text
旅行后复盘 -> 提炼个人偏好 -> 本地记忆库 -> 下次规划前检索 -> 个性化路线生成
```

报告主题建议：

> 基于 RAG 的个人旅行记忆增强：让「路线改改鸭」从一次性攻略生成器，升级为会复盘、会学习偏好的旅行助手。

这个方向可以明确体现两点：

- 会用 AI：使用 LLM 做复盘总结、偏好抽取、路线规划上下文增强。
- 有自己的思考：关注 AI 如何从用户真实旅行经验中学习，而不是只生成通用攻略。

## 2. 个人增量边界

### 2.1 做什么

第一版只做一个可演示、可截图、可写报告的轻量 RAG 实验版：

1. 用户可以录入旅行复盘文本。
2. 系统调用 LLM 把复盘抽取成结构化旅行记忆。
3. 记忆保存到本地 JSON 文件。
4. 生成或修改路线时，可以选择「使用旅行记忆」。
5. 系统检索相关记忆，把历史偏好注入现有规划 Prompt。
6. 输出结果中保留本次使用的记忆摘要，方便报告对比。

### 2.2 暂不做什么

为了避免课程作业过度工程化，第一版暂不做：

- 向量数据库。
- 用户账号和云同步。
- 照片自动识别与相册接入。
- 多人协同记忆。
- LangChain / LlamaIndex 等重框架。
- 复杂权限系统。

第一版使用本地 JSON + 简单检索 + Prompt 注入即可说明 RAG 思想。

## 3. 产品流程设计

### 3.1 旅行后复盘流程

```text
用户完成一次旅行
  ↓
填写复盘文本
  - 去了哪里
  - 哪些地方喜欢
  - 哪些地方踩坑
  - 实际花费
  - 体力感受
  - 同行人
  - 下次注意事项
  ↓
LLM 抽取结构化偏好
  ↓
保存到 output/travel_memory.json
```

示例输入：

```text
这次杭州两日游整体比较舒服。最喜欢西湖边慢慢走和小巷里的本地小吃，
不太喜欢排队很久的网红餐厅。第二天安排了 6 个点，下午明显有点累。
下次类似城市旅行每天 3-4 个主要地点就够了，交通最好在同一区域内串起来。
```

示例输出：

```json
{
  "id": "2026-06-21-hangzhou-review",
  "trip_title": "杭州两日游复盘",
  "destination": "杭州",
  "companions": "朋友",
  "liked": ["城市漫步", "湖边风景", "本地小吃", "节奏宽松"],
  "disliked": ["排队网红店", "一天超过5个景点", "频繁换乘"],
  "pace_preference": "每天3-4个主要地点最舒适",
  "budget_preference": "愿意为体验付费，不愿为高价网红餐买单",
  "traffic_preference": "优先同一区域步行和地铁串联，避免跨城区折返",
  "lessons": [
    "短视频推荐适合激发兴趣，但需要结合距离和体力重新筛选",
    "下午应留出休息时间"
  ],
  "source_text": "原始复盘文本..."
}
```

### 3.2 旅行前规划流程

```text
用户上传视频并填写新旅行需求
  ↓
可选：开启「使用旅行记忆」
  ↓
根据目的地、主题、强度、预算、补充偏好检索历史记忆
  ↓
形成 personal_memory_context
  ↓
注入现有 run_pipeline / revise_plan Prompt
  ↓
生成更贴近用户历史偏好的路线
```

示例记忆上下文：

```text
以下是用户过往旅行复盘中与本次规划相关的个人偏好，请优先遵守：
1. 用户偏好城市漫步、本地小吃、节奏宽松。
2. 用户不喜欢排队网红店、频繁换乘、一天超过 5 个景点。
3. 用户更适合每天 3-4 个主要地点，下午保留休息或低强度活动。
4. 交通上优先同一区域串联，避免跨城区折返。
```

## 4. 技术设计

### 4.1 新增文件

建议新增：

```text
travel_memory.py
```

职责：

- 读取和写入 `output/travel_memory.json`。
- 调用 LLM 把复盘文本抽取为结构化记忆。
- 根据本次旅行请求检索相关记忆。
- 生成可注入 Prompt 的 `personal_memory_context`。

### 4.2 本地存储

新增文件：

```text
output/travel_memory.json
```

结构：

```json
{
  "version": 1,
  "memories": [
    {
      "id": "2026-06-21-hangzhou-review",
      "created_at": "2026-06-21T00:00:00-07:00",
      "trip_title": "杭州两日游复盘",
      "destination": "杭州",
      "companions": "朋友",
      "liked": ["城市漫步", "本地小吃"],
      "disliked": ["排队网红店", "频繁换乘"],
      "pace_preference": "每天3-4个主要地点最舒适",
      "budget_preference": "愿意为体验付费，不愿为高价网红餐买单",
      "traffic_preference": "优先同一区域步行和地铁串联",
      "lessons": ["下午应留出休息时间"],
      "source_text": "原始复盘文本"
    }
  ]
}
```

### 4.3 检索策略

第一版不使用向量库，采用可解释的轻量检索：

1. 拼接查询文本：
   - 目的地
   - 天数
   - 预算
   - 强度
   - 主题
   - 补充偏好
2. 对每条记忆计算关键词命中分：
   - destination 命中加权最高。
   - liked / disliked / lessons 命中加权较高。
   - companions / pace / traffic 命中中等。
3. 取 Top 3 记忆。
4. 生成一段自然语言记忆摘要注入 Prompt。

优点：

- 不新增复杂依赖。
- 检索逻辑容易解释，适合课程报告。
- 出错时容易手动检查。

可选升级：

- 后续可把每条记忆转成 embedding，用向量相似度检索。
- 报告中可把这作为未来工作，而不是第一版必须完成。

### 4.4 API 设计

新增接口：

#### `GET /api/memory`

返回所有旅行记忆。

#### `POST /api/memory/reflect`

输入一次旅行复盘，调用 LLM 抽取结构化记忆并保存。

请求：

```json
{
  "review_text": "这次杭州两日游...",
  "api_key": "sk-...",
  "base_url": "https://api.openai-next.com",
  "model": "gpt-4o-mini"
}
```

响应：

```json
{
  "ok": true,
  "memory": {
    "trip_title": "杭州两日游复盘",
    "liked": ["城市漫步", "本地小吃"]
  }
}
```

#### `POST /api/memory/retrieve`

用于调试和报告展示：输入本次旅行需求，返回命中的历史记忆和摘要。

请求：

```json
{
  "destination": "成都",
  "days": 3,
  "budget": 800,
  "travel_style": "轻松型",
  "themes": ["美食", "城市漫步"],
  "extra": "想轻松一点，多吃东西"
}
```

响应：

```json
{
  "ok": true,
  "context": "用户偏好城市漫步、本地小吃、节奏宽松...",
  "matches": [
    {
      "id": "2026-06-21-hangzhou-review",
      "score": 8,
      "trip_title": "杭州两日游复盘"
    }
  ]
}
```

### 4.5 接入现有生成流程

修改 `/api/generate`：

- 新增请求字段：

```json
{
  "use_memory": true
}
```

- 如果开启记忆：
  - 后端检索相关记忆。
  - 把记忆摘要合并进现有 `extra` 字段，或新增参数传入 `run_pipeline`。

第一版建议使用最小侵入方式：

```text
extra = extra + "\n\n【个人旅行记忆】\n" + personal_memory_context
```

这样可以少改 `pipeline.py` 的函数签名。

修改 `/api/revise`：

- 新增请求字段：

```json
{
  "use_memory": true
}
```

- 如果开启记忆：
  - 把个人旅行记忆加入修改诉求：

```text
用户要求：...

请同时参考以下个人旅行记忆：
...
```

### 4.6 输出记录

为了方便报告截图和对比，生成后的 `plan.summary` 建议增加：

```json
{
  "memory_context": "本次参考了用户过往偏好：城市漫步、本地小吃、每天3-4个点...",
  "memory_matches": [
    {"trip_title": "杭州两日游复盘", "score": 8}
  ]
}
```

前端可以在底部或侧栏展示：

```text
本次参考的旅行记忆：
- 杭州两日游复盘：偏好城市漫步、本地小吃，避免排队网红店
```

## 6. 报告实验设计

### 6.1 实验问题

> 加入旅行记忆后，AI 生成的旅行路线是否更符合用户长期偏好？

### 6.2 实验材料

准备 2-3 条复盘样本：

1. 杭州两日游：喜欢城市漫步和小吃，不喜欢排队网红店。
2. 重庆三日游：喜欢夜景和本地街区，但不喜欢连续爬坡和过密行程。
3. 苏州一日游：喜欢园林和安静街巷，不喜欢商业化步行街。

### 6.3 对比方法

用同一个新需求生成两版路线：

```text
需求：成都 3 日游，轻松一点，多吃东西，预算适中。
```

对比：

| 对比项 | 普通 AI 规划 | RAG 记忆增强规划 |
|---|---|---|
| 景点数量 | 是否过密 | 是否控制在 3-4 个点 |
| 餐饮选择 | 是否偏网红 | 是否偏本地小吃 |
| 交通安排 | 是否跨区折返 | 是否同区串联 |
| 避雷提醒 | 是否泛泛而谈 | 是否继承历史踩坑 |
| 个性化程度 | 像通用攻略 | 像认识用户 |

## 8. 验证计划

### 8.1 基础验证

- 无记忆文件时，现有路线生成不受影响。
- 记忆文件为空时，开启 `use_memory` 不报错。
- 保存复盘后，`GET /api/memory` 能返回记录。
- 检索接口能返回相关记忆摘要。

### 8.2 功能验证

- 同一需求在不开启记忆时生成普通路线。
- 开启记忆后，Prompt 中包含历史偏好。
- 生成结果中能看到 `memory_context`。
- 一句话修改路线时也能参考旅行记忆。

### 8.3 报告验证

- 至少完成一组 A/B 对比。
- 对比结果能说明路线强度、餐饮偏好、交通安排或避雷提醒发生了合理变化。

## 9. 风险与取舍

### 9.1 RAG 不等于记忆一定正确

历史偏好可能过时。比如用户过去不喜欢爬山，但这次可能专门想徒步。因此 UI 要保留开关，并允许用户补充当次偏好覆盖历史记忆。

### 9.2 过度个性化会降低探索性

如果 AI 总是迎合过去偏好，可能让旅行越来越同质化。报告中可以提出改进：保留 1-2 个「轻度探索点」，并解释为什么值得尝试。

### 9.3 隐私问题

旅行复盘可能包含个人行踪、消费、同行人信息。第一版只保存在本地 JSON，不上传到额外服务；但调用 LLM 抽取时仍会发送复盘文本，需要在报告里说明隐私边界。

### 9.4 第一版检索不是严格向量 RAG

第一版采用关键词检索，是轻量 RAG-like 实现。优点是简单、可解释、便于课程演示。未来可以升级为 embedding 检索。

## 10. 期末报告建议结构

### 10.1 小组项目概要

- 项目名称：路线改改鸭。
- 核心问题：短视频旅行信息碎片化，难以转成可执行攻略。
- 小组方案：多模态视频解析 + LLM 路线规划 + 地图可视化 + 一句话改路线。

### 10.2 个人增量问题

- 一次性攻略生成无法积累用户经验。
- 用户真正需要的是越来越懂自己的旅行助手。
- 因此引入旅行后复盘和旅行前 RAG 记忆检索。

### 10.3 方法设计

- 旅行复盘结构化。
- 本地旅行记忆库。
- 规划前检索相关偏好。
- 将记忆作为上下文注入路线生成。

### 10.4 实验与结果

- 准备历史旅行复盘样本。
- 对同一目的地生成普通版和记忆增强版。
- 从路线强度、景点选择、交通安排、避雷提醒、个性化程度五个维度对比。

### 10.5 个人反思

- AI 擅长把零散经验结构化。
- RAG 能让路线规划更贴近个人经验。
- 但历史偏好不应完全替代当次探索目标。
- 旅行记忆涉及隐私，需要本地存储和可控开关。

## 11. 推荐结论表述

可以在报告结尾写：

> 小组项目解决了「如何把短视频内容转化为可执行旅行路线」的问题；我的个人增量进一步探索了「AI 如何从用户真实旅行经验中学习偏好，并在未来规划中复用这些经验」。通过旅行后复盘、结构化记忆抽取和规划前检索注入，路线改改鸭从一次性生成工具变成了具备个人记忆雏形的旅行助手。这个实践说明，RAG 的价值不只是补充外部知识，也可以帮助 AI 建立用户自己的经验上下文。
